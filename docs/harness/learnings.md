# Harness Learnings

Este arquivo registra aprendizados técnicos do projeto que devem orientar futuros agentes.

### 2026-06-30 - Slice 7-M1: Outbox multi-instancia usa FOR UPDATE SKIP LOCKED + sweep de Processing orfao; cobertura E2E 498 testes

- Contexto: Ate a Slice 7-M1 (2026-06-30), o `OutboxDispatcherWorker` rodava em single-instance: `OutboxRepository.GetPendingForDispatchAsync` retornava rows `Pending` sem lock transacional, e o Worker chamava `MarkProcessing()` + `eventStore.SaveAsync()` em iteracoes separadas (race window). Alem disso, o estado `Processing` podia ficar orfao para sempre apos crash/restart do Worker, com zero recovery automatico. Os gaps `M1-security` (sweep automatico) e `C.3-qa` (SKIP LOCKED) estavam documentados como pendentes de Phase 7 desde a auditoria de 2026-06-17.
- Decisao: (1) **Claim transacional com `FOR UPDATE SKIP LOCKED`**: `OutboxRepository.ClaimPendingForDispatchAsync(int batchSize, DateTime now, ct)` abre `BeginTransactionAsync(IsolationLevel.ReadCommitted)` no `NpgsqlConnection` raw (obtido via `_db.Database.GetDbConnection()`; NAO disposed porque EF Core owns the connection; fechado apenas se estava fechado antes via `connectionWasClosed` flag), executa `SELECT id FROM outbox_events WHERE status='Pending' AND (next_retry_at IS NULL OR next_retry_at <= @now) ORDER BY created_at FOR UPDATE SKIP LOCKED LIMIT @batchSize`, depois `UPDATE outbox_events SET status='Processing', processing_started_at=@now WHERE id = ANY(@claimedIds)`, comita, e devolve as entidades recarregadas via EF Core com `AsNoTracking` (evita colisao de cache com o mesmo row no mesmo DbContext; o worker re-attaches via `IOutboxEventStore.SaveAsync`). EF Core 10 nao traduz `SKIP LOCKED` em LINQ, entao raw ADO.NET e' obrigatorio. **(2) Sweep de Processing orfao**: `SweepOrphanedProcessingAsync(DateTime cutoff, ct)` faz um unico `ExecuteSqlRawAsync` (NAO `ExecuteSqlInterpolated`, risco de SQL injection) com parametros nomeados `@now`/`@cutoff`. Filtro obrigatorio: `WHERE status='Processing' AND processing_started_at IS NOT NULL AND processing_started_at < @cutoff`. **`last_error` recebe apenas o literal `'ProcessingOrphaned'`** (hardcoded no template SQL; nunca `ex.Message`, URL, blob, stack). **`next_retry_at = NULL`** (NAO `@now`) para que a row seja imediatamente re-disparavel na mesma iteracao do Worker — se gravarmos `@now`, a comparacao `next_retry_at <= @now` no claim poderia falhar por microsegundos. **(3) Worker NAO chama `MarkProcessing` separado**: o claim ja entrega rows em `Processing` com `processing_startedAt` stampado. Reintroduzir `MarkProcessing` + `SaveAsync` separado re-abre o race window. **(4) Sanity-check no claim**: Worker verifica `Status == Processing && ProcessingStartedAt != null` e pula com `LogError` rows em estado invalido (anti-regressao contra remocao futura acidental do UPDATE do claim). **(5) `OutboxEvent.ProcessingStartedAt` e' `timestamptz NULL`** (timestamp, NAO `jsonb`); limpo em toda saida de `Processing` (`MarkSent`, `MarkRetryWithCategory/Status`, `MarkFailedWithCategory/Status`, `RequeueOrphaned`). **(6) Migration `20260630184619_AddOutboxProcessingStartedAtAndIndexes`** substitui `(status, next_retry_at)` por `(status, next_retry_at, created_at)` e adiciona partial `(status, processing_started_at) WHERE status='Processing'`. **(7) Suite E2E cresceu para 498 testes** (467 unit + 31 integration): 2 novos em `OutboxDispatcherConcurrencyTests` (P1.1: 2 workers concorrentes, 1 evento, `CallCount = 1`; P1.2: 10 eventos, 3 workers, `CallCount = 10`) e 5 novos em `OutboxProcessingSweepTests` (P1.3: requeue de Processing de 2h atras; P1.4: Processing de 1s atras preservado; P1.5: Sent/Failed nao reabertos; P2.1: NextRetryAt futuro respeitado; P2.2: batch=3 respeitado em 10 eventos).
- Evidencia: 2 arquivos novos (testes E2E), 11 arquivos produtivos modificados (Domain `OutboxEvent`, Domain `WebhookDispatcherCategory`, Application `IOutboxPublisher`, Infrastructure `EntityConfigurations` + `Repositories` + `PaymentHubOptions`, Worker `OutboxDispatcherWorker`, Migration nova + ModelSnapshot regenerado). Build limpo em 9 projetos. Suite previa: 491 testes (apos Slice 7-IT). Suite nova: **498 testes** (+7). Filtros `~OutboxDispatcherConcurrency` 2 verdes, `~OutboxProcessingSweep` 5 verdes, `~OutboxDispatcher` 17 verdes, `~EndToEnd` 14 verdes (7 Slice 7-IT + 2 Slice 7-M1 concurrency + 5 Slice 7-M1 sweep). `scripts/agent-architecture-check.sh` e `scripts/agent-docs-check.sh` verdes (Worker continua sem depender de Api). `git diff --check` limpo. Report em `docs/audits/slice-7-m1-outbox-multi-instance-report-2026-06-30.md`.
- Impacto para proximos agentes: (a) **Para implementar um outbox-like (retry queue, scheduled job queue, fanout queue) em Postgres** copiar este pattern: (1) interface `I<Domain>Repository` com `<Verb>` que retorna `IReadOnlyList<T>` (claim) + `Sweep<State>` (recovery); (2) implementacao com raw `NpgsqlConnection` + `BeginTransactionAsync` + `SELECT ... FOR UPDATE SKIP LOCKED LIMIT @batchSize` + atomic UPDATE; (3) entidade domain com `ProcessingStartedAt` (timestamp nullable) + `MarkProcessing(DateTime now)`; (4) Worker com sanity-check no claim. (b) **Para sweep automatico de qualquer estado transitorio**: usar `ExecuteSqlRawAsync` com parametros nomeados (NAO `ExecuteSqlInterpolated`); template SQL estavel para `EXPLAIN`; `last_error`/`<recovery_marker>` recebe apenas literal hardcoded da categoria (`'ProcessingOrphaned'`, `'RetryOrphaned'`, etc.) — nunca `ex.Message`. (c) **Em testes de concorrencia** usar `Task.WhenAll` com workers instanciados via `factory.Services.GetRequiredService<>`; cada worker usa seu proprio `IServiceScope` + DbContext (EF Core DbContext nao e' thread-safe; compartilhar mascara bugs reais). Anti-flaky: `await Task.Delay(10)` entre os `DispatchOnceAsync` se o teste flake-ar. (d) **Ao adicionar nova coluna de timestamp no schema**: usar `timestamptz NULL`, NAO `jsonb`. Nao confundir: timestamp e' numero, JSON e' documento. (e) **`outbox_events.payload` permanece `jsonb`**: conteudo controlado pelo PaymentHub, indexado em queries internas. (f) **`webhook_events.raw_payload` permanece `text`**: byte-exact HMAC preservation (Slice 3-IT BLOCKER). (g) **`provider_accounts.webhook_events` permanece `text`**: byte-exact round-trip (Slice 2-C BLOCKER). (h) **Migracao nova NAO toca colunas `payload`/`raw_payload`/`webhook_events`**: re-asserting Slices 7-IT, 3-IT, 2-C. (i) **Workaround anti-regressao para `EF Core` + `SKIP LOCKED`**: raw `NpgsqlConnection` e' o unico caminho. EF Core 10 nao traduz em LINQ.

### 2026-06-30 - Slice 2-C descobre o mesmo anti-pattern jsonb de Slice 3-IT em nova coluna, opta por text + re-aproveita inspector pattern para HasWebhookSecret

- Contexto: Slice 2-C (2026-06-30) adiciona 4 colunas non-sensitive a `provider_accounts` para gerenciar a inscricao de webhook AbacatePay via API. O plano original marcava `webhook_events` como `jsonb` ("para queries SQL/GIN-index"), exatamente como o Slice 3-IT ja tinha feito (e revertido) para `webhook_events.raw_payload`. Apos a primeira execucao do teste `ProviderAccount_ShouldPersistAllWebhookConfigurationColumns`, a string retornada tinha um espaco extra apos cada `,` e `:` ("differs near t" no erro de FluentAssertions), confirmando que **Postgres `jsonb` reformata JSON no insert**, mesmo quando o caller envia um payload ja serializado em C#. Alem disso, o handler precisava responder `hasWebhookSecret: bool` em GET sem nunca expor o valor, replicando o padrao que `ProcessWebhookEventHandler.ExtractWebhookSecret` ja usava para resolver o segredo no caminho de webhook inbound.
- Decisao: (1) **Trocar `provider_accounts.webhook_events` para `text`** (mesma regra que `webhook_events.raw_payload`): o Postgres `jsonb` normaliza o JSON no insert (insere espaco apos cada `:`, `,`), o que quebraria qualquer round-trip byte-exact ou HMAC no body armazenado. Como `WebhookEvents` e usado apenas pelo parser de resposta (que desserializa em memoria via `JsonDocument.Parse`), `text` e o tipo correto. Migration `20260630001726_AddProviderAccountWebhookColumns` corrigida. Anti-regression note espelhada no comentario inline em `EntityConfigurations.cs` e na migration. (2) **Extrair `ProviderAccountCredentialsInspector`** (helper `internal static` em `Application/Tenants/Validation/`, mesmo pattern de `WebhookUrlValidator`) para o caminho de "unprotect + parse JSON + decidir se existe webhookSecret". O inspector e puro (sem DI, sem logging, sem exceptions). O handler e o GET response builder usam-no em vez de duplicar a logica de `ProcessWebhookEventHandler.ExtractWebhookSecret`. (3) **NUNCA usar `jsonb` para armazenar JSON** que o sistema insere via `JsonSerializer.Serialize(...)` e reconsome via `JsonDocument.Parse(...)`. O padrao "INSERT byte-exact + RECONSTROY in-memory" e o caso classico de `text`. Reservar `jsonb` apenas quando o SQL precisa consultar campos do JSON (GIN index, predicates). (4) **DTO de response nao expoe secret-shaped fields** (validado por reflexao em 3 testes distintos): nenhum getter chamado `ApiKey`, `WebhookSecret`, `ProtectedWebhookSecret`, `EncryptedCredentials`. O unico acesso ao segredo pelo consumidor da API e o boolean `hasWebhookSecret`. (5) **Feature flag opt-in + interface provider-agnostic em Application**: `Providers:AbacatePay:AllowWebhookRegistration` (default `false`) + `IProviderWebhookRegistrationFeaturePolicy` em `Application/Abstractions/Providers/`. Implementacao concreta em `Infrastructure.Providers.AbacatePay/` (`AbacatePayWebhookRegistrationFeaturePolicy`) + no-op `NoOpProviderWebhookManagementClient`. Tudo isso para que Application NAO dependa de Infrastructure (Clean Architecture) e para que o gate de "remote registration" seja sempre opt-in ate o Slice 2-C.1 substituir o no-op pelo client HTTP real.
- Evidencia: migration nova `20260630001726_AddProviderAccountWebhookColumns` (4 AddColumn, todos `text`/`varchar`/`timestamp`, zero `jsonb`). Migration Designer + ModelSnapshot regenerados pelo `dotnet ef migrations remove/add` (com fallback manual quando o tool nao conecta ao DB). Suite previa: 422 testes. Suite nova: **484 testes (+59 unit + 3 integration)**, todos passando. `dotnet build PaymentHub.slnx` 0/0 (9 projetos). `dotnet test PaymentHub.slnx` em ~6s. `dotnet test PaymentHub.IntegrationTests.csproj` 17 passando (14 Slice 1-IT + 3 Slice 2-C). Filtros `~AbacatePay`, `~ProviderAccount`, `~Webhook`, `~EndToEnd` verdes. `scripts/agent-architecture-check.sh` passou (Application continua sem depender de Infrastructure). Relatorio em `docs/audits/slice-2c-abacatepay-webhook-management-report-2026-06-30.md`. Specs `006`, `008`, `009`, `011` atualizadas com a nova secao. Validation matrix extendida com bloco Phase 2 Slice 2-C.
- Impacto para proximos agentes: (a) **Postgres `jsonb` reformata JSON no insert** — sempre que armazenar corpo bruto que precisa ser validado por HMAC, assinatura digital, hash ou checksum externo, OU que precisa ser reconsumido em memoria via `JsonDocument.Parse`, usar `text` (ou `json`, que preserva o original). `jsonb` so quando o payload precisa de queries SQL/GIN-index. **ANTI-PATTERN (DO NOT)**: reverter `provider_accounts.webhook_events` para `jsonb` em nova migration; o teste `ProviderAccountWebhookPersistenceTests.ProviderAccount_ShouldPersistAllWebhookConfigurationColumns` quebra com diff de whitespace. **ANTI-PATTERN (DO NOT)**: adicionar novas colunas `*_json` marcadas como `jsonb` para "conveniencia" — auditar sempre a politica de tipo de coluna quando uma migration nova tocar JSON-shaped data. (b) **Sempre usar `internal static` + `InternalsVisibleTo("PaymentHub.UnitTests")` para helpers puros de validacao/inspecao** (`ProviderAccountCredentialsInspector`, `WebhookUrlValidator`, etc.). Manter essa convencao evita duplicacao de logica entre `ProcessWebhookEventHandler.ExtractWebhookSecret` e os novos handlers de PUT/GET. (c) **Application-layer abstractions for provider-specific behavior**: a Slice 2-C demonstrou que e possivel ter `IProviderWebhookManagementClient` + `IProviderWebhookRegistrationFeaturePolicy` em `Application/Abstractions/Providers/` mesmo quando a implementacao e AbacatePay-specific. A implementacao concreta vive em `Infrastructure.Providers.AbacatePay/` e a feature flag segue a configuracao do provider concreto. **ANTI-PATTERN (DO NOT)**: importar `PaymentHub.Infrastructure.*` em `PaymentHub.Application` (quebra `scripts/agent-architecture-check.sh`). (d) **DTOs de webhook management NAO aceitam `tenantId`/`applicationId`** — re-asserting Slice 6-B. Endpoint autenticado server-to-server SEMPRE deriva esses ids de `ITenantContext` via try/catch em `InvalidOperationException` retornando 401. (e) **Campos configuracionais (callbackUrl, eventos, status) ficam em colunas own-column non-sensitive**. `webhookSecret` continua dentro de `EncryptedCredentials`. Regras de hash/criptografia + cleanup aplicados identicamente aos 2 paths (inbound webhook via `ProcessWebhookEventHandler.ExtractWebhookSecret`; outbound config via `ProviderAccountCredentialsInspector.HasWebhookSecret`).

### 2026-06-29 - Slice 2-B: AbacatePay webhooks externos usam verifier puro + normalizer puro + routing por metadata + fail-fast 401 no controller

- Contexto: Ate o Slice 2-B (2026-06-29), o `AbacatePayProviderAdapter.ParseWebhookAsync` ainda era o parsing JSON naive implementado para o Slice 2-A (fazia match de `data.id`/`data.status` e retornava `IsValid=true` sem verificar HMAC nem validar evento). (3) **Roteamento por metadata do payload**: `ProcessWebhookEventHandler.ResolveAbacatePayWebhookSecretAsync` faz parse permissivo de `data.metadata.{tenantId, applicationId, paymentId}` (tolerante a JSON malformado, devolve `null` em vez de lancar) e usa `IPaymentRepository.GetByIdForTenantAsync(tenantId, paymentId)` para localizar o pagamento no escopo do tenant. ### 2026-06-25 - Webhook secret protection usa AES reversivel com prefixo de proposito, parametrizacao explicita e chave separada de credenciais

- Contexto: O Slice 6-C enderecou o gap P1-5 (`ApplicationClient.WebhookSecret` persistido em texto claro). ### 2026-06-27 - Slice 2-A: AbacatePay transparente PIX sandbox usa resolver-as-value-object, IOptionsMonitor + named HttpClient + Singleton client, e safe-envelope pattern

- Contexto: Ate o Slice 2-A (2026-06-27), `AbacatePayProviderAdapter` ainda era skeleton (`Success=false` hard-coded). `IPaymentProviderAdapter` era Singleton e nao podia receber repository scoped; `CreateCheckoutHandler.ResolveProviderAsync` resolvia `ProviderAccount` para validar existencia e descartava a entidade, impossibilitando o adapter de receber credenciais. `CreateCheckoutProviderRequest` nao carregava dados de credencial. `PaymentStatusMapper.MapAbacatePay` existia mas sem testes explicitos para os status da doc oficial (`PENDING`, `PAID`, `EXPIRED`, `CANCELLED`, `FAILED`, `REFUNDED`, `REDEEMED`, `UNDER_DISPUTE`). - Decisão: (1) **Resolver-as-value-object**: `ResolveProviderAsync` passou a retornar um record `internal ResolvedProvider` com `ProviderCode`, `ProviderAccountId`, `Environment`, `EncryptedCredentials`. (3) **Pattern outbound typed client + IOptionsMonitor + IHttpClientFactory nomeado**: `IAbacatePayClient` depende apenas de `IHttpClientFactory` (Singleton-safe), `IOptionsMonitor<AbacatePayOptions>` (Singleton-safe) e `ILogger<>` (Singleton-safe). (4) **Bearer Token via `IHttpClientFactory`**: `request.Headers.Authorization = new AuthenticationHeaderValue("Bearer", apiKey)`. A chave nunca e persistida nem logada; o adapter chama `ICredentialProtector.Unprotect` uma unica vez e o JSON desprotegido (`{apiKey, secret}`) e descartado apos extrair o `apiKey`. (5) **Safe-envelope + categorized-error pattern**: `AbacatePayEnvelope<T>` modela `{ data, success, error? }`. `AbacatePayClient` valida `success=true` antes de retornar e lanca `AbacatePayClientException(EnvelopeFailure)` caso contrario. Categorias: `BadRequest`, `Unauthorized`, `NotFound`, `RateLimited`, `ServerError`, `Network`, `Timeout`, `EnvelopeFailure`, `Unexpected`, `SimulationDisabled`. `IsTransient` e derivado da categoria (true para `RateLimited`/`ServerError`/`Network`/`Timeout`). **Mensagem da exception NAO inclui API key, body, brCodeBase64, response body** — so categoria + statusCode opcional. Caller `CancellationToken` cancelado propaga como `OperationCanceledException`, distinguindo cancelamento do operador de timeout do `HttpClient`. (6) **Simulate gated by config**: `AllowDevModeSimulation` (default `false` em `appsettings.json`, `true` em `appsettings.Development.json`); o client lanca `SimulationDisabled` **antes** de montar request quando flag false — preferencia explicita por "falhar cedo". (7) **`BaseAddress` path gotcha**: `BaseAddress = new Uri(baseUrl.TrimEnd('/') + "/")` com path `/transparents/create` (leading slash) faz `HttpClient` resolver do host root, **ignorando** `/v2/`. Correcao: caminhos **sem** leading `/` (ex.: `transparents/create`), preservando o segmento `/v2/` do `BaseAddress`. (8) **Status mapping estendido com decisao documentada**: `redeemed → Approved` (PIX ja resgatado conta como pago final), `under_dispute → Pending` (decisao explicita: ate existir anti-fraude chargeback, mantemos intermedio). Testes cobrem todos os 11 status do mapping mais case-insensitive do provider code. (9) **`AesCredentialProtector` constructor exige configuration**: testes nao podem instanciar diretamente sem `PaymentHub:CredentialEncryptionKey`. Solucao: `FakeCredentialProtector` em `tests/Support/` (mesmo pattern de `FakeWebhookSecretProtector`) com marker prefix reversivel, **nao criptografico**, usado apenas em testes de adapter.
